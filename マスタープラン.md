# sticky マスタープラン v1.0

## 1. 目的
本書は、`sticky` の開発を `全体方針 -> フェーズ管理 -> フェーズ別計画書 -> 実装` の順で進めるための親計画である。  
今後の実装は、本書を最上位の進行管理文書として参照し、各フェーズの着手前に個別計画書を作成する。

本書の役割は以下の3つ。

1. 全体の流れを固定する
2. 現在どのフェーズにいるかを明確にする
3. 各フェーズの計画書に必要な前提条件を定める

---

## 2. 最上位方針
`sticky` は、思考の初速を落とさず、その場で書いて閉じ、後から回収できる Mac 向けメモアプリである。  
初期MVPでは、以下を絶対方針とする。

- macOS 専用
- ローカル完結
- プレーンテキスト
- セッション型メモ管理
- 高度機能より、入力速度と操作の自然さを優先
- Tauri v2 + React + SQLite を第一候補として進める
- 仕様変更や実装着手は、必ず計画書を先に作る

## 4. フェーズ管理の基本ルール
今後の実装は、以下の流れを必須とする。

1. マスタープランで対象フェーズを確認する
2. フェーズ別計画書を作成する
3. 独立レビューまたはセルフチェックを行う
4. Gate 条件を満たしたら、その計画書に従って実装する
5. 実装後、結果と差分を計画書または関連文書に反映する

禁止事項:
- 計画書なしでの直接実装
- 複数フェーズの目的を同時に進めること
- フェーズ計画書を作らずに「ついで実装」を広げること

### Phase 2: デスクトップメモの最小成立
目的:
- 1枚セッションを中心に、メモを開く・選ぶ・編集する・保存する最小体験を成立させる

対象:
- `MemoCard` 本実装
- シングルクリック選択
- ダブルクリック / Enter 編集
- テキスト入力
- `Cmd + S`
- `Cmd + Enter`
- 1枚セッション表示
- スロット配置
- 基本リサイズ

完了条件:
- 1枚セッションを開いて書いて保存して閉じられる
- 表示・選択・編集が破綻しない

備考:
- このフェーズでは管理画面や複数セッション運用を広げすぎない

---

### Phase 3: セッション操作の成立
目的:
- 複数メモ・複数セッションを前提とした sticky 独自の操作モデルを成立させる

対象:
- 複数枚セッション開始
- セッション色割り当て
- セッション選択導線
- 右クリックメニュー
- セッション単位削除
- セッション単位の閉じる
- セッション単位の auto close

完了条件:
- セッション単位の論理構造とデスクトップ操作が一致している
- 1枚セッションと複数枚セッションが同じ保存モデルで扱える

---

### Phase 4: 保存基盤と自動処理
目的:
- ローカル保存を本実装し、autosave / cleanup / 再起動耐性を成立させる

対象:
- SQLite 永続化
- memo / session 保存
- autosave
- title 再生成
- is_open 管理
- 起動時 cleanup
- 空メモ非表示

完了条件:
- 保存経路が一本化されている
- autosave と明示保存で差異がない
- クラッシュ後の再起動でも整合性が保てる

---

### Phase 5: 管理画面 MVP
目的:
- 保存済み内容を確認・回収・再表示できるようにする

対象:
- Home
- Trash
- Settings
- キーワード検索
- メモ再表示
- セッション移動
- 削除 / 復元 / 完全削除

完了条件:
- 管理画面から sticky の保存内容を運用できる
- 「閉じたら終わり」ではなく「後で回収できる」が成立する

---

### Phase 6: 操作感・UX安定化
目的:

- autosave / auto close の体感調整

完了条件:
- 日常使用に耐える
- 主要操作で大きなストレスがない

## 8. フェーズ別計画書の必須構成
各フェーズ計画書には最低限以下を含める。

1. SSOT参照宣言
2. 今回触る関連ファイル
3. 問題一覧（ID付き）
4. 修正フェーズ / 実装ステップ
5. Gate条件
6. 回帰 / 副作用チェック
7. セルフチェック結果
8. 変更履歴

補足:
- 詳細は `AI-Planning-Guidelines-Sticky.md` に従う
- 本書は「どのフェーズをいつ作るか」を管理し、個別計画書は「そのフェーズをどう実装するか」を定義する

---

## 9. フェーズ着手条件
各フェーズは、前段フェーズの Gate 条件を満たしてから着手する。

例外:
- 緊急修正
- 疎通確認のためのプローブ
- 文書整備だけを先に行う場合

ただし、例外時も以下は必要。
- なぜ順序を飛ばすのか
- 何を最小確認するのか
- 後でどのフェーズに戻すのか

---

## 10. Gate 管理の考え方
Gate 条件は「主観的に良さそう」ではなく、「次フェーズへ進めるだけの前提が揃ったか」で判定する。

例:
- Phase 1 の Gate:
  - 透過前面レイヤーの初期疎通ができている
  - グローバルショートカットが起動する
  - Tauri で進行不能という結論ではない

- Phase 2 の Gate:
  - 1枚セッションで入力・保存・閉じるが成立する
  - 選択と編集の切替が明確

---

## 11. 運用ルール
- 実装開始前に、対象フェーズ計画書を必ず作る
- 実装中に仕様変更が出た場合、先に文書を更新する
- 実装完了後は、必要に応じてマスタープランの `現状` と `現在位置` を更新する
- 1フェーズ完了ごとに、次フェーズへ進む前にレビューする

## 13. 結論
今後の sticky 開発は、以下の順で進める。

`マスタープラン確認 -> フェーズ計画書作成 -> レビュー/セルフチェック -> 実装 -> 文書反映`

この流れを固定することで、AI 特有の文脈放棄・仕様飛び・ついで実装を防ぎ、  
sticky の開発を段階的かつ保守的に進める。
